# 活动模块需求文档

> **版本**: v1.0  
> **作者**: Jack  
> **同步来源**: [[surge-server]]`surge-mvp/doc/activity-module.md`

## 一、需求概述

实现活动管理全流程功能，涵盖 **Admin 管理后台**、**APP 端（用户端）**、**H5 微信端**、**H5 WhatsApp 端**四个客户端，支持活动的创建、发布、报名、签到、分享传播链追踪等完整业务闭环。

## 四、活动状态管理

### 4.1 状态定义

| 状态码 | 状态 | 说明 |
|--------|------|------|
| 0 | 草稿 | 创建后默认状态，不对外展示 |
| 1 | 已发布/待开始 | 已发布但活动时间未到 |
| 2 | 进行中 | 当前时间在 startTime 和 endTime 之间 |
| 3 | 已结束 | 当前时间超过 endTime |
| 4 | 已取消 | 手动取消，不可修改 |

### 4.2 状态动态修正

查询列表和详情时，根据当前时间动态计算真实状态（仅对非草稿、非取消的活动）：

```java
if (endTime < now)       → status = 3  // 已结束
else if (startTime > now) → status = 1  // 待开始
else                      → status = 2  // 进行中
```

修正后同步更新数据库 `status` 字段。

### 4.3 列表排序

按状态优先级排序：**进行中(2) > 待开始(1) > 已结束(3)**，同状态按创建时间倒序。

### 4.4 列表过滤

- 排除草稿（status=0）和已取消（status=4）
- 排除下架活动（shelf_status=0），兼容 null（视为上架）
- Tab 筛选按时间条件查询而非 status 字段

## 五、报名逻辑

### 5.1 报名校验

| 校验项 | 说明 |
|--------|------|
| 活动存在性 | 活动不存在返回失败 |
| 活动状态 | 已结束(3)或已取消(4)不可报名 |
| 名额限制 | quotaType=1 时，registrationCount >= quota 不可报名 |
| 重复报名 | 同一 userId + activityId 已有 status!=3 的记录不可重复报名 |

### 5.2 报名数据存储

- `formData` 格式：`[{"name":"张三"},{"phone":"138xxx"},{"email":"xxx@qq.com"},...]`
- 报名时从 formData 提取 `name` → `userName`、`phone` → `userPhone`
- 从 `blade_user` 获取 `avatar` → `userAvatar`
- 初始化 `status=1`（已报名）、`checkinStatus=0`（未签到）

### 5.3 前端表单校验（根据 Admin 配置动态控制）

| 字段 | fieldKey | 校验规则 |
|------|----------|---------|
| 姓名 | name | isRequired=1 时必填，不超过20字符 |
| 手机号 | phone | isRequired=1 时必填，支持大陆11位、香港8位(5-9开头)、澳门8位(6开头) |
| 电子邮箱 | email | isRequired=1 时必填，填写时校验邮箱格式 |
| 性别 | gender | isRequired=1 时必选 |
| 出生日期 | birthDate | isRequired=1 时必选 |
| 证件类型 | idType | isRequired=1 时必选（居民身份证/护照/港澳通行证） |
| 证件号码 | idNumber | isRequired=1 时必填，选择居民身份证时校验18位格式 |
| 备注 | remark | 选填，最多200字 |

字段显示/隐藏和必填/选填由 Admin 后台 `surge_activity_form_field` 表的 `isVisible` 和 `isRequired` 控制。

## 六、签到逻辑

| 场景 | 处理方式 |
|------|---------|
| 活动已结束/已取消 | 不可签到 |
| 已签到 | 不可重复签到 |
| 报名+签到模式 | 必须先报名才能签到 |
| 仅签到模式 | 未报名也可签到，自动创建参与记录 |

## 八、分享与传播链

### 8.1 分享流程（APP 端）

1. 用户点击"邀请报名" → 调传播链接口 `blade-instrument/wxArticles/trackingMaterial`
   - `materialType: '1'`（MaterialConstant.MATERIAL_TYPE_ACTIVITY）
2. 获取链路参数：browsingId、chain、directAgentId、parentPropagationId、senderId、shareTime
3. 拼装分享链接 → 弹出分享面板（复用 shareNewsDia 组件）
4. 支持：微信好友、朋友圈、QQ、WhatsApp、复制链接、系统分享

### 8.2 分享链接格式

**微信端**：
```
${shareUrl}activity/?id=${activityId}&browsingId=xxx&chain=xxx&directAgentId=xxx&parentPropagationId=xxx&senderId=xxx&shareTime=xxx
```

**签到二维码**：
```
${shareUrl}activity/checkin/?id=${activityId}&browsingId=xxx&chain=xxx&...
```

**WhatsApp 端**：
```
${shareUrl}whatsApp?params=Base64({materialId, materialType:1, chain, teamUserId, clientUserId})
```

### 8.3 传播链兼容

`checkMaterialExists` 方法（blade-instrument 服务）新增 `MATERIAL_TYPE_ACTIVITY(1)` 支持，活动类型直接放行（活动数据在 surge-mvp 服务）。

## 九、Long 精度丢失处理

所有实体类的 Long 类型 ID 字段加 `@JsonSerialize(using = ToStringSerializer.class)` 注解：

- `Activity.id`
- `UserActivity.id`、`activityId`、`userId`
- `ActivityFormField.id`、`activityId`

Controller 的 `@PathVariable` 统一用 `String` 接收，内部 `Long.parseLong()` 转换。

## 十一、参与者详情接口说明

### 接口：`GET /app/activity/{activityId}/participant-detail?userId=xxx`

根据 `activityId + userId` 查询 `surge_user_activity` 表，解析 `formData` JSON：

**返回格式**：
```json
{
  "userActivity": { ... },
  "fieldValues": [
    { "fieldKey": "name", "fieldLabel": "客户姓名", "value": "张三" },
    { "fieldKey": "phone", "fieldLabel": "手机号", "value": "13800138000" },
    { "fieldKey": "email", "fieldLabel": "电子邮箱", "value": "test@qq.com" },
    { "fieldKey": "gender", "fieldLabel": "性别", "value": "男" },
    { "fieldKey": "birthDate", "fieldLabel": "出生日期", "value": "1990-01-01" },
    { "fieldKey": "idType", "fieldLabel": "证件类型", "value": "居民身份证" },
    { "fieldKey": "idNumber", "fieldLabel": "证件号码", "value": "440106xxxx" },
    { "fieldKey": "remark", "fieldLabel": "备注", "value": "" }
  ]
}
```

固定 8 个字段按顺序返回，从 `formData` 数组 `[{key:value}]` 合并为 Map 后按 fieldKey 取值。

## 十三、注意事项

1. **formData 格式**：数据库存储为 JSON 数组 `[{key:value}]`，不是 Map，Entity 类型为 `List<Map<String, Object>>`
2. **状态动态修正**：查询时根据时间动态计算，同时回写数据库
3. **Long 精度**：所有 ID 字段 JSON 序列化为 String，Controller 用 String 接收
4. **权限**：H5 接口需在 Nacos 配置 `blade.secure.skip-url` 放行
5. **传播链**：活动 materialType=1，需在 `checkMaterialExists` 中放行
6. **上下架**：shelf_status 默认为 1（上架），null 视为上架，下架后 APP 端不可见
